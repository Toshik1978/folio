# Ebook Parsing

> Per-format metadata & cover extraction in `internal/ebook`, plus the
> annotation rendering pipeline.
>
> **Status:** Implemented. The dispatcher and the EPUB / FB2 / MOBI-AZW3 / PDF
> parsers are coded and tested against committed fixtures under
> `internal/ebook/testdata/` (generated by `gen.go`).

---

## Overview

`internal/ebook` turns a single book file into an `ebook.Metadata` value
(title, authors, series + series number, language, annotation, publisher, year,
genres, identifiers, page count, and a cover image). A dispatcher picks the
parser by extension (`.epub`, `.fb2`, `.fb2.zip`, `.mobi`, `.azw`, `.azw3`,
`.pdf`); each format has its own quirks documented below. The `.fb2.zip`
wrapper normalizes to the `fb2` format label (see the FB2 section).

The package depends only on the standard library plus `pdfcpu` and
`golang.org/x/text` (legacy charset decoding — FB2 declared encodings via
`encoding/ianaindex`, MOBI cp1252 via `encoding/charmap`), all pure-Go, with a
sanitizer at the serve boundary — no CGo — so the binary still builds statically
(`CGO_ENABLED=0`) for Alpine/Distroless. See
[BUILD-AND-DEPLOY.md](./BUILD-AND-DEPLOY.md).

### Metadata model

`ebook.Metadata` carries a `SeriesNumber float64` (0 = unknown) alongside the
`Series` name, so the series **index** survives parsing — not just the name.
`recordFromMeta` threads it into `bookRecord.SeriesNumber`, and the ingest
gap-fill stores it on the logical book (see
[DATABASE.md](./DATABASE.md#metadata-merge--gap-fill)).

## Shared rules

### Plausible-year floor

`ParseYear` (`internal/ebook/ebook.go`) accepts a 4-digit year only when it
falls in `[minPlausibleYear, 9999]` with `minPlausibleYear = 1000`. This
rejects leading-zero / sentinel dates such as `"0101-01-01"` (which would
otherwise parse as year `101`). Every parser — including Calibre — routes
through it, so there is no per-importer special case.

### Series index

`parseSeriesIndex` (`internal/ebook/ebook.go`) parses positions like `"2"` or
`"1.5"`, returning 0 when absent. It is fed by the FB2 `<sequence number>`
attribute and the EPUB `calibre:series_index` meta (see per-format notes).

## EPUB

- **Series number** — read from `<meta name="calibre:series_index" content="N">`
  via `parseSeriesIndex`.
- **Genres** — every `<dc:subject>` is extracted as a candidate tag. These are
  free text ("Science Fiction"); at ingest they are matched against the
  controlled genre taxonomy and any subject that doesn't resolve to a canonical
  label is **intentionally discarded** (see [Genre taxonomy](#genre-taxonomy)).
- EPUBs already emit HTML descriptions (`<dc:description>`), so their
  annotations need no markup translation — only the serve-time sanitizer.
- EPUB is the **highest-priority** format for cross-edition metadata/cover
  merges (see [format priority](./DATABASE.md#format-priority)).

## FB2

- **`.fb2.zip` wrapper** — zipped FB2 files are handled by `parseFB2Zip`
  (`internal/ebook/fb2.go`): it opens the archive, parses the first `.fb2`
  entry it finds, and errors when the archive contains none. `DetectFormat`
  normalizes the `.fb2.zip` extension to the plain `fb2` format label, so the
  stored `book_files.file_format` — and the served Content-Type — match a bare
  FB2. Fixture: `testdata/test.fb2.zip`.
- **Series number** — read from the FB2 `<sequence name="…" number="N"/>`
  attribute.
- **Genres** — every `<genre>` is extracted as a flibusta code (`sf_history`,
  `prose_classic`). At ingest these codes are mapped to readable canonical labels
  via the genre taxonomy; the same map also covers the INPX `inpGenre` field.
  Codes with no mapping are discarded (see [Genre taxonomy](#genre-taxonomy)).
- **Annotation normalization** — FB2 uses its own markup vocabulary, not HTML,
  so a raw FB2 `<annotation>` would be flattened by the serve-time sanitizer.
  `parseFB2XML` runs the annotation through `normalizeFB2Annotation`
  (`internal/ebook/fb2.go`), which rewrites the FB2 vocabulary into the safe
  HTML subset the sanitizer keeps:

  | FB2 tag | HTML |
  | :--- | :--- |
  | `<emphasis>` | `<em>` |
  | `<strikethrough>` | `<s>` |
  | `<empty-line/>` | `<br/>` |
  | `<strong>` `<sub>` `<sup>` `<code>` `<p>` | preserved |
  | `<a xlink:href>` | `<a href>` |

  Unknown tags are dropped while their text is kept; character data / entities
  are resolved and re-escaped as HTML via the XML decoder. Malformed input is
  returned unchanged — the serve-time sanitizer is the backstop.

## MOBI / AZW3 (KF8)

The MOBI parser (`internal/ebook/mobi.go`) reads the PalmDB record 0 (PalmDOC +
MOBI header) and the EXTH block. Several producer-specific quirks are handled:

- **Title — prefer EXTH 503 over the header title.** Retail KindleGen files
  often store an entity-encoded and truncated record-0 header title (e.g.
  `Death&#x2019;s End`), while EXTH record 503 ("updated title") holds the
  clean, full title (`Death's End (Remembrance of Earth's Past)`). The parser
  prefers 503 when present and `html.UnescapeString`s text fields, so entities
  decode and the full title/subtitle survives. Calibre builds write a clean
  header title that matches 503, so there is no regression.
- **Cover — `firstImageIndex` lives at MOBI header offset 92.** That is
  `rec0[108:112]` from the start of record 0 (PalmDOC header 16 bytes + MOBI
  offset 0x5C). An earlier bug read offset 76 (the Language field), yielding a
  bogus image base and a wrong-image fallback. With the correct base,
  `firstImageIndex + coverOffset` (EXTH 201) locates the real cover for both
  MOBI6 and KF8.
- **Annotation — EXTH 103 is the only canonical description record.** When a
  file has no record 103 the description simply isn't in that file (confirmed:
  retail KindleGen `Death's End.azw3` lacks 103 while the Calibre build of the
  same book has it). This is a **data-availability** gap, not a parser defect —
  re-running the parser can't help. The remedy is **cross-edition annotation
  sharing**: a sibling format (e.g. the EPUB's `<dc:description>`) supplies the
  description via ingest gap-fill. See
  [DATABASE.md](./DATABASE.md#metadata-merge--gap-fill).

## PDF

- **Cover** — `parsePDF` validates + optimizes the cross-reference table, then
  extracts the first embedded image on **page 1** via pdfcpu
  (`coverFromContext` → `ExtractPageImages`). Digital books almost always embed
  their cover as the page-1 image, so this avoids a heavy vector-rendering or
  CGo dependency. Image kinds that are never covers (stencil masks, page
  thumbnails) are skipped. Scanned PDFs whose page 1 is the cover work; born-PDF
  with no page-1 image simply get the placeholder.
- PDFs carry plain-text descriptions only, so the serve-time sanitizer suffices.

## Metadata coverage by format

What each parser can populate (✓ = yes, — = not available in the format):

| Field | EPUB | FB2 | MOBI / AZW3 | PDF |
| :--- | :---: | :---: | :---: | :---: |
| Title | ✓ | ✓ | ✓ (EXTH 503 ▸ header) | ✓ |
| Authors | ✓ | ✓ | ✓ | ✓ |
| Genres | ✓ (`dc:subject`, free text) | ✓ (`<genre>`, flibusta codes) | — | — |
| Annotation | ✓ (HTML) | ✓ (normalized) | ✓ (EXTH 103, often absent) | — |
| Series | ✓ | ✓ | — | — |
| Series number | ✓ (`calibre:series_index`) | ✓ (`<sequence number>`) | — | — |
| Language | ✓ | ✓ | ✓ | maybe |
| Publisher | ✓ (`dc:publisher`) | ✓ (`publish-info`) | ✓ (EXTH 101) | — |
| Year | ✓ (`dc:date`) | ✓ (`publish-info`) | ✓ (EXTH 106) | ✓ (`CreationDate`) |
| Pages | — | — | — | ✓ |
| Identifiers | ✓ (`dc:identifier`) | ✓ (ISBN) | ✓ (EXTH 104 ISBN) | — |
| Cover | ✓ (manifest) | ✓ (base64) | ✓ (EXTH 201 + `firstImageIndex`) | ✓ (page-1 image) |

The Calibre and INPX importers don't parse files directly — they read the index
(Calibre `metadata.db`, INPX records), which carries title/authors/series/etc.
Fields missing from an edition stay NULL and may be gap-filled from a sibling
format (see [DATABASE.md](./DATABASE.md#metadata-merge--gap-fill)).

Two further tiers fill what the index and sibling formats can't, lazily on first
view: a **local backfill** parses the actual book file
(`ingest.Extractor.Backfill`, used for INPX books — it recovers the annotation
and identifiers the index lacks), and an **online enrichment** tier queries
Google Books for the metadata-poor formats above — PDFs (no
annotation/genres/identifiers) and retail MOBIs (often no EXTH 103 description)
especially, matching by ISBN or title+author. Both gap-fill, are guarded, and run
at most once; see [DATABASE.md](./DATABASE.md#lazy-metadata-tiers-on-view).

## Genre taxonomy

Raw genre strings from every source are folded into a small, curated set of
canonical labels by `normalizeGenres` (`internal/ingest/genres.go`) before they
are linked to a book — on both the insert and the overwrite merge path. The
taxonomy has two parts:

- **`fb2GenreMap`** — maps raw flibusta/INPX codes (`sf_history`,
  `prose_classic`, …) to a canonical label (`Alternative History`, `Classics`).
- **`targetGenres`** — the whitelist of canonical labels. A raw string that is
  already a canonical label (case-insensitive) passes through unchanged.

**The canonical labels are BISAC subject labels** (`Science Fiction`, `Crime`,
`Mystery & Detective`, `Biography & Autobiography`, …). That single choice is
what makes the three sources converge:

- **Google Books** returns BISAC-derived category *paths* (`"Fiction / Science
  Fiction / Space Opera"`). `normalizeRawTag` matches the whole string first,
  then — failing that — splits on `/` and matches each segment, so a path still
  contributes its recognized segment(s) (here, `Science Fiction`). Aligning the
  whitelist to BISAC means enrichment fills the same shelves verbatim instead of
  dropping unmatched labels.
- **Calibre** tags survive by design when the library is itself BISAC-tagged,
  because each tag *is* a canonical label — Calibre tags run through the same
  `normalizeGenres` on import, so a non-BISAC tag would be silently dropped.
- **FB2 / INPX** codes are mapped to their BISAC label by `fb2GenreMap`.

Two intentional **non-BISAC** buckets are kept: `Nonfiction` (the catch-all;
BISAC has no such subject) and `Contemporary Fiction` (BISAC only has
"Contemporary" nested under specific genres). Neither is produced by Google
enrichment — they are FB2/manual-only — but both are heavily used shelves.

A string that matches **neither** map is **discarded by design** — the taxonomy
is deliberately curated rather than open-vocabulary.

The overwrite merge path re-links genres from the highest-priority edition, so a
book's stored genres are always the normalized set of its current metadata owner.
`relinkGenres` no-ops rather than wiping when nothing maps, so a higher-priority
edition without recognized genres can't erase the existing set. To recognise more
inputs, extend `fb2GenreMap` (new code→label) or `targetGenres` (new canonical
label — keep it a valid BISAC subject so Google enrichment can match it).

## Identifier normalization & links

Raw identifiers from every source (file parsers, Calibre, INPX, online
enrichment) are cleaned and deduped by `cleanIdentifier` / `cleanIdentifiers`
(`internal/ingest/identifier.go`) before they reach `book_identifiers`. Like the
genre taxonomy, this is a single-sourced cleaning pass — sync ingestion, the lazy
backfill, and Google Books enrichment all route through it
(`cleanedEbookIdentifiers`), so an identifier lands identically regardless of
which path produced it.

The rules:

- **Filter junk schemes** — drop the type entirely for `uuid`, `urn`, `uri`,
  `calibre`, `mobi-uid`, `local`.
- **Filter UUID-like values** — drop `urn:uuid:` / `urn:calibre:` prefixed values
  and anything that parses as a UUID (with or without a `uuid:` prefix).
- **Fold types to a canonical key** — `amazon-asin` / `mobi-asin` / `asin` →
  `amazon`; `isbn-10` / `isbn-13` / `isbn10` / `isbn13` / `isbn_10` / `isbn_13` →
  `isbn` (the `_10` / `_13` forms are Google Books' `IndustryIdentifiers` type
  strings). Folding to a shared key lets the `UNIQUE(book_id, type)` upsert
  collapse duplicates to one row.
- **Normalize values** — ISBNs lose any `urn:isbn:` / `isbn:` prefix, hyphens, and
  spaces and are upper-cased; ASINs are upper-cased.
- **Dedupe per type** — when several values fold to the same type, keep the best
  one: ISBN-13 beats ISBN-10 beats other lengths (`lookupISBN`); for other types
  the longer (more informative) value wins.

Because the persisted identifiers are the cleaned set, they also feed
`content_hash` — junk-only source edits (a dropped calibre UUID, ISBN
punctuation) don't change the fingerprint and so don't trigger a spurious
metadata refresh (see [DATABASE.md](./DATABASE.md#metadata-merge--gap-fill)).

### Provider links

At the serve boundary, `identifierURL` (`internal/api/bookview.go`) maps a cleaned
identifier type to an outgoing link the SPA renders on the book detail (a type
with no mapping renders as plain text, no link):

| Type | Link |
| :--- | :--- |
| `isbn` | `https://isbnsearch.org/isbn/{value}` |
| `amazon` | `https://www.amazon.com/dp/{value}` |
| `goodreads` | `https://www.goodreads.com/book/show/{value}` |
| `google` | `https://books.google.com/books?id={value}` |

The Google Books `google` type is supplied by online enrichment / Fix Match (the
volume id), so an enriched book always gets at least one canonical link.

## Annotation rendering pipeline

Annotations are rendered in the SPA via `v-html`
(`web/src/components/BookDetail.vue`, the book-detail modal body), so they must
be **sanitized** to prevent XSS. Sanitizing happens at the **serve boundary**
(not at ingest), plus a defense-in-depth pass in the client — three layers in
total:

- `toBookView` (`internal/api/bookview.go`) runs every annotation through
  `bluemonday.UGCPolicy()` before it leaves the REST API.
- The OPDS catalog sanitizes the same way: `bookEntry`
  (`internal/opds/feeds.go`) runs the annotation through its own
  `bluemonday.UGCPolicy()` before placing it in `<content type="html">`, so the
  HTML reaching reader apps is sanitized too.
- The SPA re-sanitizes client-side: `BookDetail.vue` runs the annotation
  through **DOMPurify** (`DOMPurify.sanitize`) before binding it to `v-html`.
  This is deliberate defense-in-depth — the browser never trusts the payload
  even if a server-side pass is bypassed or regresses. Covered by
  `BookDetail.spec.ts`.
- Serve-time was chosen over ingest-time because it protects **every** source
  and any already-stored data immediately, with no re-ingest needed.

Sources differ in what they store, so each is shaped to land inside the policy's
allowed HTML subset:

| Source | Annotation form | Handling |
| :--- | :--- | :--- |
| EPUB / Calibre / MOBI | already HTML | sanitize only |
| FB2 | FB2 markup vocabulary | `normalizeFB2Annotation` at ingest → sanitize |
| PDF | plain text | sanitize only |
| Google Books (enrichment) | HTML/plain description | sanitize only (serve-time pass covers it) |

## Tests

`internal/ebook` follows the package's testify-suite layout (see
[BACKEND.md](./BACKEND.md#test-strategy)): a `TestEbook` runner plus per-format
suites (`epubSuite`, `fb2Suite`, `mobiSuite`, `pdfSuite`) against committed
`testdata/` fixtures. Notable cases: FB2 tag mapping
(`TestNormalizeAnnotationMapsFB2Tags` / `…FallsBackOnMalformed`), MOBI title
preference (`TestUpdatedTitlePreferredOverHeader`), KF8 cover
(`TestKF8CoverIsFirstImage`), series index (`fb2Suite.TestSeriesNumber` /
`epubSuite.TestSeriesIndex`), and the year-floor sentinels.
